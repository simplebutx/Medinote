## 프로젝트 소개

**MediNote**는 처방전 OCR, 약 정보 RAG 챗봇, 복약 일정 관리, 복약 기록 추적 기능을 제공하는 AI 기반 복약 관리 서비스입니다.

사용자가 처방전 이미지를 업로드하면 AI 서버가 복약 정보를 추출하고, 백엔드는 해당 결과를 복약 일정 도메인과 연결하여 개인화된 복약 관리 흐름을 제공합니다.

---

## 핵심 설계 포인트

* OCR 결과를 단순 텍스트 추출에 그치지 않고 복약 일정 생성 흐름과 연결
* 사용자 복약 일정 및 복약 기록 데이터를 챗봇 컨텍스트로 활용
* 처방전 인식 → 복약 일정 생성 → 복약 기록 → 약 정보 질문 응답으로 이어지는 순환 구조 설계
* AI 기능을 서비스 외부 부가 기능이 아닌 복약 관리 도메인의 핵심 흐름으로 통합

---

## 담당 역할

AI 기능 연동과 복약 도메인 백엔드 구현을 중심으로 담당했습니다.

* FastAPI 기반 AI 서버 구성
* 처방전 OCR 파이프라인 구현
* Qdrant 기반 약 정보 RAG 챗봇 구현
* OCR 결과 기반 복약 일정 자동 생성 흐름 연동
* 복약 일정, 복약 시간, 복약 기록 관리 API 설계 및 구현
* 사용자 복약 데이터를 챗봇 컨텍스트로 제공하는 백엔드 연동 구조 구현

---

## 주요 기여

### AI 서버 및 OCR 파이프라인

* FastAPI 기반 AI 서버 구성 및 백엔드 연동 API 설계
* 처방전 이미지에서 약 이름, 복용량, 복용 횟수, 복용 기간 등 복약 정보 추출
* 이미지 품질 평가, 회전 보정, OCR confidence 요약 로직 추가
* OCR 결과를 구조화된 JSON 형태로 변환하여 복약 일정 생성에 활용
* 다양한 처방전 이미지 입력에서도 분석 가능한 OCR 처리 흐름 구성

### 약 정보 RAG 챗봇

* Qdrant 벡터 DB와 임베딩 모델을 활용한 약 정보 검색 기반 RAG 챗봇 구현
* 사용자가 특정 약을 언급하거나 복약 일정 관련 질문을 입력하면 약 정보와 사용자 복약 컨텍스트를 함께 활용하도록 구성
* 단순 LLM 응답이 아닌 검색된 문서 기반 답변 구조 설계
* 답변 유형을 일반 답변, fallback, 약사 상담 권장으로 구분하여 안전한 응답 흐름 구성
* 의료 정보 답변에서 근거 부족 또는 위험 가능성이 있는 경우 상담 권장 흐름으로 분리

### 복약 관리 백엔드

* Spring Boot 기반 복약 서비스에서 OCR 결과를 복약 일정 도메인과 연결
* 복약 일정, 복약 시간, 복약 기록을 관리하는 도메인 모델 및 CRUD API 설계
* 사용자 복약 일정 데이터를 챗봇 컨텍스트로 제공하는 연동 구조 구현
* 약 정보 검색, 복약 주의 태그, 처방 분석 기능을 AI 서버와 백엔드 서비스 간 API 호출 구조로 연결

## 문제 해결 경험

### 다양한 약봉투 양식에 대응하는 OCR 후처리

약봉투마다 표 구조와 헤더 표현이 달라 하나의 고정된 파싱 규칙으로는 안정적인 정보 추출이 어려웠습니다.  
이를 해결하기 위해 여러 추출 로직을 동시에 실행하고, 각 결과를 점수화하여 가장 신뢰도 높은 결과를 최종 선택하는 방식으로 후처리 구조를 개선했습니다.  
또한 새로운 양식이 추가되더라도 전체 로직을 수정하지 않고 추출 패턴만 확장할 수 있도록 코드를 분리했습니다.

### OCR 모델 비교를 통한 기술 선택

OCR 품질이 복약 일정 자동완성의 정확도에 직접적인 영향을 주기 때문에 Google Vision, Naver OCR, Azure OCR을 비교했습니다.  
핵심 키워드 추출 여부, 노이즈 수, 응답 시간, 글자 오류율을 기준으로 평가했고, Azure가 비용과 노이즈 측면에서 장점이 있음을 확인했습니다.  
다만 기존 후처리 로직과의 호환성 및 교체 비용을 고려해 현재는 Google Vision을 유지하고, 추후 운영 규모가 커질 경우 Azure 전환을 검토하는 방향으로 결정했습니다.

### RAG 챗봇 검색 품질 개선

약 관련 질문은 사용자 표현과 의약품 문서 표현이 다를 수 있어 단순 키워드 검색만으로는 정확한 근거 문서를 찾기 어려웠습니다.  
이를 해결하기 위해 질문에서 핵심 키워드와 유사 표현을 함께 추출하고, 키워드 매칭 점수와 벡터 유사도 점수를 함께 사용하여 문서 청크의 관련도를 계산했습니다.  
이를 통해 "술", "알코올", "음주"처럼 표현이 달라도 같은 의도에 해당하는 문서를 검색할 수 있도록 개선했습니다.

### 임베딩 및 LLM 모델 비교

RAG 검색 품질을 높이기 위해 OpenAI small, Solar Embedding, OpenAI large 임베딩 모델을 비교했습니다.  
Solar Embedding이 가장 높은 검색 정확도를 보였지만, 과금 정책과 사용량 제한을 고려해 최종적으로는 품질과 비용의 균형이 좋은 OpenAI large를 선택했습니다.

LLM 역시 단순 답변 품질보다 의료 정보 서비스에 필요한 근거 기반 응답, 규칙 준수, 위험 상황에서의 상담 안내 여부를 중심으로 비교했습니다.  
그 결과 GPT-4o mini가 비용 효율과 안정적인 답변 품질을 모두 만족한다고 판단하여 최종 적용했습니다.
